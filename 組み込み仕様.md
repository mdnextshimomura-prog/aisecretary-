# 組み込み仕様書 — AI秘書を親エージェント配下に差し込むために

最終更新: 2026-07-01

このドキュメントは、**親側（統括エージェント）の担当者がそのまま読んで、このAI秘書を配下に
組み込むため**の接続宣言です。
- 子としての中身・制約 … `CLAUDE.md`
- 移行手順・設計の意図 … `引き継ぎ書.md`
- 親への接続のしかた（このファイル）… 下記

---

## 0. 一行宣言
> 「不動産業務のメッセージを、タスクとしてNotionに登録し、期日をLINEでリマインドする」
> 単機能・完結型の秘書。**一旦この内容のまま“子”として差し込める。**

## 1. 階層上の位置づけ（子か孫か）

**設計方針：位置ではなく契約に対して作ってある。** 下記の入出力契約さえ守れば、
置き場所は配線だけで決まり、中身の書き換えは不要。

- **既定：子（親の直下）**
  ```
  親（統括秘書） ─▶ 子：不動産タスク秘書（このapp）
  ```
  親は「不動産まわりのタスク登録・リマインドはこの子に任せる」と振り分けるだけ。

- **必要なら：孫（別の子の下にぶら下げる）**
  親の下にすでに「不動産業務」全般を持つ子がいるなら、このappはその一機能＝孫になる。
  ```
  親（統括秘書） ─▶ 子：不動産業務秘書 ─▶ 孫：タスク登録・リマインド（このapp）
  ```
  **この変更で書き換わるのは“誰が呼ぶか（配線）”だけ。** 契約も実装もそのまま。

## 2. 担当ドメインの境界（重要）
| 区分 | 内容 |
|---|---|
| ✅ 引き受ける | LINEメッセージ → タスク構造化 → Notion登録 → 期日リマインド／**LINE指示によるメール下書き・確認・Gmail送信** |
| ⤴ 親／他の子へ返す（スコープ外） | 物件調査・重要事項説明など書類作成、LINE以外の入力チャネル（メール受信等）、タスクの実行判断そのもの |

この境界を越える依頼が来たら、このエージェントは扱わず、親に差し戻す前提。

## 3. インターフェイス契約（親からの呼び方）
このappは**デプロイ済みのHTTPサービス**（Next.js on Vercel）。親からの接続は2通り選べる。

### 経路A：自走サービスとして“見守る”（既定・推奨）
親は直接呼ばない。LINE Webhook経由で自律稼働している。親は
「このドメインは子が自動で処理している」と認識し、結果（Notionの登録状況）だけ参照する。
- 親が知っておくこと：**担当ドメイン**と**Notion DBの所在**のみ。

### 経路B：ツールとして“呼び出す”
親や他の子から能動的に使わせたい場合、HTTPエンドポイントをツールとして渡す。

| 用途 | 入力 | 出力 |
|---|---|---|
| タスク登録 | `{ message: string, today: "YYYY-MM-DD" }` | `{ isTask, confidence, title, category, urgency, dueDate, assignee }` ＋ Notion登録 |
| 期日近接タスク取得 | （なし） | `[{ id, title, dueDate, urgency, assignee, url }]` |

- 内部処理：`src/lib/claude.ts` の `parseTaskFromMessage()` が判定＋解析、
  `confidence >= TASK_CONFIDENCE_THRESHOLD`（既定0.7）で `src/lib/notion.ts` が登録。
- ※ 経路Bで使う場合は、親が叩ける薄いAPIルート（例：`/api/tasks` の入力受け口）を
  用意すると接続が楽。現状は `/api/webhook`（LINE専用・署名検証あり）が主入口。

## 4. 親が用意・確認するもの（接続の前提）
- このappの**環境変数**が新環境に設定済みであること（→ `引き継ぎ書.md` 第4章）
- LINE Webhook URL がこのappのドメインに向いていること
- Notion インテグレーションが対象DBにアクセスできること
- （経路Bを使うなら）親が叩く入力APIの口を1本追加

## 5. 子→孫を増やす余地（将来）
このapp自身も、内部でClaude API（`claude.ts`）を“部品”として呼んでいる。
さらに専門処理（例：物件名の正規化、重複検知）を分けたくなったら、
それらを孫として切り出せる。**その場合も本仕様の対外契約（第2〜3章）は変えない。**
