# 引き継ぎ書 — AI秘書（LINE秘書）

最終更新: 2026-06-30

このドキュメントは、本プロジェクトを **親・子・孫構造のエージェントの「子エージェント」** として
別環境へ引き継ぐための説明書です。技術的な役割定義は `CLAUDE.md` を、
背景・意図・移行手順はこの文書を参照してください。

---

## 1. これは何か（一言で）
不動産業務のLINEメッセージを、Claudeが解析してNotionタスクに自動登録し、
期日が近づいたらLINEでリマインドする「秘書」アプリ。Next.js 14 + Vercel構成。

## 2. なぜこの設計になったか（判断の経緯）
コミット履歴に現れた設計判断を、意図とともに残します。

- **永続化はNotionを主軸にした**
  当初はSQLite（Prisma）とGoogleカレンダーを併用していたが、運用上Notionで
  一元管理する方が現場（人間）が確認しやすいため、Notionを主軸に切り替えた。
  SQLiteはリマインド処理（`remind.ts`）に、Googleカレンダー連携（`calendar.ts`）は
  将来の再利用に備えて**残置**してある（現フローでは未使用）。

- **「メンションされた時だけ」反応する**
  グループLINEの全発言に反応すると誤登録が多発するため、ボットへの
  `@メンション` がある発言だけをタスク化する仕様にした。

- **担当者はメンションのテキストから直接取得**
  LINEのメンバー名取得APIは権限・取得失敗のケースがあったため、
  メッセージ内のメンション文字列から名前を直接拾う方式に変更した（最新版）。

## 3. 子エージェントとしての位置づけ
- **役割**：「LINE入力 → タスク構造化 → Notion登録 → リマインド」を完結させる単機能の子。
- **境界**：不動産書類作成・他チャネル入力・タスクの実行判断は**この子の責務外**。
  親が振り分けるか、別の子（例: `realestate-*` 系スキル）に委譲する。
- **入出力契約**：`CLAUDE.md` 末尾「親エージェントへの返却契約」を参照。

## 4. 引き継ぎ方法（重要）

### 結論：「ワークスペース移行」だけでは完了しない。下記3点を必ずセットで行う。

**(A) ワークスペースを移行する**
git リポジトリごと（または親リポジトリのサブディレクトリ／サブモジュールとして）移す。
これでコード・スキーマ・このドキュメント・`CLAUDE.md` が運ばれる。

**(B) 環境変数を手動で再設定する ← gitでは運ばれない**
`.env.local` は `.gitignore` 除外のため移行されない。新環境（特にVercel）で
以下を再設定する必要がある：
```
ANTHROPIC_API_KEY, LINE_CHANNEL_ACCESS_TOKEN, LINE_CHANNEL_SECRET,
LINE_BOT_USER_ID, NOTION_API_KEY, NOTION_DATABASE_ID
（任意）GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, GOOGLE_REFRESH_TOKEN, GOOGLE_CALENDAR_ID
```

**(C) 外部サービス側の接続を確認する ← コードの外にある状態**
- LINE Developers：Webhook URLを新環境のドメイン（`/api/webhook`）に向け直す。
- Notion：インテグレーションが対象DBに招待されているか、プロパティ名（日本語）が一致しているか。
- Vercel：リマインド用のScheduled Function / Cronが新環境で有効か。

### なぜ「移行だけ」で済まないのか
| 運びたいもの | gitで運べる？ | 補完手段 |
|---|---|---|
| ソースコード・DB設計 | ✅ | (A)で完了 |
| 秘密鍵（.env.local） | ❌ 除外 | (B)で手動再設定 |
| 設計の意図・判断理由 | △ 断片のみ | この引き継ぎ書 |
| 子としての役割・契約 | ❌ | `CLAUDE.md` |
| 外部サービスの接続状態 | ❌ | (C)で確認 |

## 5. 移行後の動作確認チェックリスト
- [ ] `.env.local`（または新環境の環境変数）を全項目設定した
- [ ] `npm install && npm run build` が通る
- [ ] LINE DevelopersのWebhook URLを新ドメインに更新した
- [ ] Notionインテグレーションが対象DBにアクセスできる
- [ ] グループでボットを`@メンション`して送信 → Notionに登録され、LINEに「✅ タスク登録しました」が返る
- [ ] リマインドのcron/Scheduledが新環境で有効

## 6. 既知の注意点・残課題
- `calendar.ts` は未使用のまま残置。完全に不要なら削除可。
- リマインドは `remind.ts` がSQLite（Prisma）を見る一方、登録はNotion主軸。
  リマインド対象もNotion参照に統一すると整合性が増す（`notion.ts` の `getUpcomingTasks` が利用可能）。
